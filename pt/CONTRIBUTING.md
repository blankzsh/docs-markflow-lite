# Guia de Contribuição

Obrigado pelo seu interesse no projeto MarkFlow Lite! Damos as boas-vindas a contribuições de qualquer tipo, seja reportando bugs, sugerindo novos recursos, melhorando a documentação ou enviando código.

## Código de Conduta

Por favor, siga nosso [Código de Conduta](CODE_OF_CONDUCT.md) para garantir que todos os participantes possam se comunicar em um ambiente aberto e amigável.

## Como Contribuir

### Reportando Bugs

Ao reportar um bug, por favor inclua as seguintes informações:

1. Um título claro e conciso
2. Passos detalhados para reproduzir o problema
3. Comportamento esperado e comportamento real
4. Informações do navegador e sistema operacional
5. Capturas de tela ou gravações relevantes (se aplicável)

### Sugerindo Recursos

Damos as boas-vindas a sugestões de novos recursos! Ao enviar uma sugestão de recurso, por favor considere o seguinte:

1. Descreva o recurso que você deseja em detalhes
2. Explique como o recurso melhoraria a experiência do usuário
3. Forneça casos de uso ou exemplos

### Melhorando a Documentação

A documentação é uma parte importante do projeto. Se você encontrar erros na documentação ou desejar adicionar novo conteúdo, sinta-se à vontade para enviar um PR.

### Enviando Código

#### Configuração do Ambiente de Desenvolvimento

1. Faça um fork do repositório do projeto
2. Clone seu fork: `git clone https://github.com/your-username/markflow-lite.git`
3. Crie um branch de recurso: `git checkout -b feature/your-feature-name`
4. Instale as dependências: `npm install`
5. Faça suas alterações
6. Execute os testes para garantir que tudo funciona corretamente
7. Commit suas alterações: `git commit -am 'Add some feature'`
8. Envie para o branch: `git push origin feature/your-feature-name`
9. Crie um Pull Request

#### Padrões de Codificação

- Siga o estilo de código existente no projeto
- Adicione comentários apropriados para explicar lógica complexa
- Certifique-se de que o código seja fácil de entender e manter
- Escreva mensagens de commit significativas

#### Diretrizes de Mensagens de Commit

Usamos o formato de mensagem de commit convencional:

- `feat: Adicionar novo recurso`
- `fix: Corrigir bug`
- `docs: Atualizar documentação`
- `style: Alterações de formatação de código (alterações que não afetam a execução do código)`
- `refactor: Refatoração (alterações de código que não são novos recursos nem correções de bugs)`
- `test: Adicionar testes`
- `chore: Alterações no processo de build ou ferramentas auxiliares`

### Processo de Pull Request

1. Certifique-se de que seu PR tenha um título e descrição claros
2. Vincule às issues relevantes (se houver)
3. Certifique-se de que todos os testes passem
4. Aguarde que os mantenedores revisem seu código
